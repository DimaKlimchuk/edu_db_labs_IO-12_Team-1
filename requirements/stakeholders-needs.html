# Запити зацікавлених осіб

# Вступ

У цьому документі міститься опис основних рис сервісу для проведення опитувань, зокрема: скорочений перелік бізнес-сценаріїв, наявна функціональність у додатку, його практичність використання, надійність опитувань, продуктивність системи та експлуатаційна придатність.

# Мета

Створення легкого у використанні та надійного сервісу, який дає можливість користуачам створювати, редагувати та аналізувати результати опитування.

# Контекст

У даному документі формується технічне завдання, метою якого є створення системи організації та управління опитуваннями експертів.

# Основні визначення та скорочення

Ділові процеси - це сукупність послідовних операцій та процедур, які виконуються програмою з метою реалізації певної бізнес-функції або завдання. Характеристики ділових процесів програми можуть бути різними в залежності від сфери застосування програмного забезпечення.[1]

Функціональність - це сукупність функцій та можливостей, які вона надає користувачу. Функціональність визначає, що програма може робити та як користувач може з нею взаємодіяти.[2]

Практичність - це здатність програмного забезпечення забезпечувати практичне застосування в реальних умовах та різних сферах діяльності.[3]

Надійність - це властивість програми, яка забезпечує її стійкість до помилок та збоїв, та здатність продовжувати свою роботу в умовах, де є помилки вводу/виводу даних, збої апаратного забезпечення, атаки зловмисників та ін.[4]

Продуктивність - це здатність програмного забезпечення виконувати свої завдання ефективно та швидко. Основні характеристики продуктивності програми можуть включати:[5]

  1. Швидкість виконання - програма повинна виконувати свої завдання швидко та без зайвих затримок. Швидкість виконання може бути особливо важливою для програм, що працюють з великими обсягами даних, а також для додатків в реальному часі.

  2. Використання ресурсів - програма повинна ефективно використовувати ресурси комп'ютера, такі як процесор, пам'ять та дисковий простір. Це дозволяє забезпечити ефективну продуктивність при мінімальному використанні ресурсів.

  3. Масштабованість - програма повинна бути здатна масштабуватися, тобто працювати з великою кількістю даних та користувачів без втрати продуктивності. Це особливо важливо для додатків, що працюють з великими обсягами даних або мають багато користувачів.

  4. Відповідність - програма повинна відповідати вимогам користувачів та бути здатною задовольняти їх потреби. Наприклад, програма може бути вимогливою щодо швидкості виконання для високопродуктивної роботи, або мати обмеження щодо ресурсів для більш економічної роботи.

Експлуатаційна придатність - це здатність програмного забезпечення бути легко зрозумілим та використовуваним користувачами з мінімальними зусиллями.[6]

# Характеристика ділових процесів

# 1. ID ПРОЦЕСУ: NEW_ACOUNT_CREATE

НАЗВА: Реєстрація нового користувача

УЧАСНИКИ: Система, користувач

ПЕРЕДУМОВИ: Користувач перейшов на сторінку, однак не має облікового запису

РЕЗУЛЬТАТ: Створено акаунт користувача

ВИКЛЮЧНІ СИТУАЦІЇ: Відміна реєстрації, обліковий запис вже існує, хибні дані

ОСНОВНИЙ СЦЕНАРІЙ:

  1. Користувач робить запит на реєстрацію
  2. Система надає форму для заповнення
  3. Користувач вводить дані
  4. Система обробляє дані
  5. Система створює акаунт користувача
  6. Система надає користивачу доступ до його акаунту

# 2. ID ПРОЦЕСУ: USER_LOGIN

НАЗВА: Вхід користувача в його акаунт

УЧАСНИКИ: Система, користувач

ПЕРЕДУМОВИ: Зареєстрований користувач перейшов на сторінку

РЕЗУЛЬТАТ: Користувачу наданий доступ до акаунту

ВИКЛЮЧНІ СИТУАЦІЇ: Відміна входу, хибні дані

ОСНОВНИЙ СЦЕНАРІЙ:

  1. Користувач вводить дані
  2. Система обробляє дані
  3. Система надає користивачу доступ до його акаунту

# 3. ID ПРОЦЕСУ: CREATE_SURVEY

НАЗВА: Створення опитування

УЧАСНИКИ: Система, користувач

ПЕРЕДУМОВИ: Користувач авторизований та знаходиться на сторінці свого особистого кабінету

РЕЗУЛЬТАТ: Створене опитування

ВИКЛЮЧНІ СИТУАЦІЇ: Користувач не створив жодного питання

ОСНОВНИЙ СЦЕНАРІЙ:

  1. Користувач переходить на сторінку створення опитування
  2. Користувач заповнює дані опитування
  3. Користувач зберігає та відправляє дані системі
  4. Система обробляє отримані дані та створює опитування
  5. Користувач завершує роботу із системою

# 4. ID ПРОЦЕСУ: UPDATE_SURVEY

НАЗВА: Редагування опитування

УЧАСНИКИ: Користувач,система

ПЕРЕДУМОВИ: Користувач авторизований та знаходиться на сторінці свого особистого кабінету

РЕЗУЛЬТАТ: Відредаговане опитування

ВИКЛЮЧНІ СИТУАЦІЇ: Користувач не зробив жодних змін

ОСНОВНИЙ СЦЕНАРІЙ:

  1. Користувач переходить на сторінку редагування опитування
  2. Користувач робить зміни та відправляє їх системі
  3. Система оновлює дані опитування
  4. Користувач завершує роботу із системою

# 5. ID ПРОЦЕСУ: DELETE_SURVEY

НАЗВА: Видалення опитування

УЧАСНИКИ: Користувач, система

ПЕРЕДУМОВИ: Користувач авторизований та знаходиться на сторінці свого особистого кабінету

РЕЗУЛЬТАТ: Видалення опитування

ВИКЛЮЧНІ СИТУАЦІЇ: Відсутні

ОСНОВНИЙ СЦЕНАРІЙ:

  1. Користувач переходить на сторінку опитування
  2. Користувач робить запит на видалення опитування
  3. Система видаляє опитування з бази даних
  4. Користувач завершує роботу із системою

# 6. ID ПРОЦЕСУ: TAKE_SURVEY

НАЗВА: Проходження опитування

УЧАСНИКИ: Користувач, система

ПЕРЕДУМОВИ: Користувач хоче проголосувати

РЕЗУЛЬТАТ: Відправка заповненої форми

ВИКЛЮЧНІ СИТУАЦІЇ: некоректно введені

ОСНОВНИЙ СЦЕНАРІЙ:

  1. Користувач переходить на сторінку опитування
  2. Користувач заповнює форму
  3. Система перевіряє отримані дані
  4. Система зберігає відповідь користувача
  5. Система відправляє заповнену форму

# 7. ID ПРОЦЕСУ: LOOK_SURVEY

НАЗВА: Перегляд доступних опитувань

УЧАСНИКИ: Користувач, система

ПЕРЕДУМОВИ: Користувач авторизований та знаходиться на головній сторінці

РЕЗУЛЬТАТ: Сторінка зі списком опитувань

ВИКЛЮЧНІ СИТУАЦІЇ: Жодне опитування недоступне

ОСНОВНИЙ СЦЕНАРІЙ:

  1. Користувач робить запит для перегляду опитувань
  2. Система знаходить доступні опитування
  3. Користувач завершує роботу із системою

# 8. ID ПРОЦЕСУ: GET_RESULT_SURVEY

НАЗВА: Отриманная результатів опитування

УЧАСНИКИ: Користувач, система

ПЕРЕДУМОВИ: Користувач знаходиться на сторінці облікового запису та опитування завершено

РЕЗУЛЬТАТ: Результати опитування

ВИКЛЮЧНІ СИТУАЦІЇ: Опитування ніхто не пройшов

ОСНОВНИЙ СЦЕНАРІЙ:

  1. Користувач переходить на сторінку опитування
  2. Користувач робить запит на отримання результатів опитування
  3. Користувач дивиться результати
  4. Користувач завершує роботу із системою

# Короткий огляд продукту

LightQuize - система управління експертними опитуваннями, яка пропонує простий та зручний для користувача інтерфейс, до того ж з широким функціоналом. З нашим додатком користувач зможе швидко та легко створювати опитування, аналізувати результати та поширювати його.

# Функціональність

Система має функціональний інтерфейс для всіх учасників робочого процесу, що дозволяє систематизувати інформацію та забезпечувати зручну взаємодію користувачів з інструментами тестувальної системи та іншими користувачами. Цей підхід дозволяє повністю реалізувати функціонал для робочого процесу користувачів системи. Інтерфейси, наведені нижче, розподіляють користувачів на певні групи, надаючи кожному відповідну роль, і потім забезпечують інструменти та функції, відповідно до цієї ролі та типу користувача.

# • Інтерфейс респондента

Респондентом є особа, яка бере участь у заповненні опитування, проходженні опитування, тестуванні або інтерв'ю за допомогою функціоналу сервісу. Він має свій обліковий запис з усіма необхідними функціями для зручного пошуку та проходження опитувань, а також для подальшої обробки отриманих результатів.

Можливості респондента :

  • Заповнювати опитування;
  • Долучатись до проходження опитувань за посиланням;
  • Здійснювати пошук доступних опитувань;
  • Надсилати заповнену анкету для обробки результатів;
  • Мати зворотний зв'язок з інтерв'юером;
  • Отримувати допомогу у технічній підтримці платформи у разі необхідності;
  • Переглядати результати надісланих опитувань.

# • Інтерфейс інтерв'юера

Інтерв'юер отримує набір інструментів для створення анкет та опитувань, а також їх подальшого редагування та модифікації. Функціональний простір також дозволяє публікувати тестування та поширювати їх серед респондентів-користувачів.

Можливості інтерв'юера:

  • Створювання нових опитувань;
  • Налаштовання окремих параметрів опитувань;
  • Змінення візуальної складової опитувань;
  • Редагування опитувань;
  • Поширення опитування.

# • Інтерфейс адміністратора

Інструменти та функціональний простір інтерфейсу адміністратора дозволяють контролювати кожен етап опитування, починаючи зі створення та розповсюдження анкети і закінчуючи збором результатів опитування та систематизацією даних.

Можливості адміністратора:

  • Видалення користувачів;
  • Передача результатів опитування інтерв'юерам;
  • Видалення опитувань;
  • Можливість зв'язку між адміністраторами та інтерв'юерами;
  • Надання зворотного зв'язку респонденту в разі непередбачуваних помилок.

# Надійність

Основні вимоги :

  • Захист від DDOS атак ;
  • Можливість обробки запитів великої кількості користувачів одночасно;
  • Резервне копіювання баз даних для запобігання втрати інформації.

# Продуктивність

Основні вимоги :

  • Максимальна можлива кількість користувачів, що одночасно проходять опитування, 1 000;
  • Здатність швидкої обробки даних та результатів;
  • Обмеження кількості запитів в секунду до 600;
  • Максимальна можлива кількість користувачів у системі 12 000.

# Експлуатаційна придатність

Основні вимоги :

  • Підтримка таких платформ як Windows, Linux, Android та IOS;
  • Проведення регулярної діагностики та виправлення помилок;
  • Використання сучасних технологій;
  • Швидке реагування на наявність помилок;
  • Прозорість коду.
